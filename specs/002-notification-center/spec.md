# Feature Specification: 通知中心（Notification Center）Phase 1 後端核心

**Feature Branch**: `r2/002-notification-center`

**Created**: 2026-06-10

**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 統一通知管線與收件匣（Priority: P1）

任何事件來源（Watchdog 警示、新資料入庫、研究任務完成…）發出一個事件，系統把它組裝成一則標準通知，送進使用者的收件匣。分析師打開收件匣能看到未讀數、依時間與嚴重度排序的列表，可標已讀、可依公司代號或通知類型篩選，每則通知都帶來源證據。

**Why this priority**: 這是整個通知中心的基底——沒有「事件進、通知出」這條管線，其他一切（合規閘門、訂閱、外送管道）都無處掛載。也是 PRD G1（分析師不用守在畫面前）的最小實現。

**Independent Test**: 餵一筆模擬事件給管線，驗證收件匣出現一則格式完整的通知（標題、摘要、嚴重度、證據、深連結），未讀數 +1；標已讀後未讀數歸零。

**Acceptance Scenarios**:

1. **Given** 空收件匣，**When** 一筆「2330 新法說稿入庫」事件進入管線，**Then** 收件匣出現 1 則通知，含標題、摘要（≤100 字）、嚴重度、至少 1 筆來源證據、回詳情的連結，且未讀數 = 1。
2. **Given** 收件匣有 3 則未讀，**When** 使用者將其中 1 則標為已讀，**Then** 未讀數變 2，且該則保留可查（不消失）。
3. **Given** 收件匣含多家公司、多種類型的通知，**When** 依公司代號（或類型）篩選，**Then** 只列出符合條件的通知。
4. **Given** 同一事件（相同事件識別碼）被重複發送兩次，**When** 兩次都進入管線，**Then** 收件匣只出現 1 則通知（exactly-once）。

---

### User Story 2 - 合規閘門：對外通知零買賣建議（Priority: P1）

每一則要送給終端使用者的通知，文案（標題＋摘要）必須先通過合規審查。命中買賣建議的通知不得派送，且系統要自動產生一則「合規事故」通知給內部團隊，留下記錄。

**Why this priority**: 憲法原則 I（NFR-031）是 No-Go 紅線。通知是新的對外輸出面——出口沒守住，前面 Workflow/Watchdog 守得再好都會破功。此 story 必須與 US1 同期交付，管線不可在無閘門狀態下上線。

**Independent Test**: 餵一筆內容誘導買進的事件，驗證收件匣**沒有**出現該通知，但內部出現一則合規事故通知。

**Acceptance Scenarios**:

1. **Given** 一筆文案會生成「建議買進」字眼的事件，**When** 進入管線，**Then** 該通知不出現在使用者收件匣，且內部收到 1 則 severity=alert 的合規事故通知（含被攔事件的識別碼）。
2. **Given** 一筆中性描述事實的事件，**When** 進入管線，**Then** 通知正常派送、合規狀態為 passed。
3. **Given** 受眾為內部（internal）的營運通知，**When** 進入管線，**Then** 不經合規攔截改寫（內部訊號非對外輸出），但仍走同一管線記錄。
4. **Given** 一筆沒有任何來源證據的對使用者事件，**When** 進入管線，**Then** 不派送（沒有來源的事件不發通知——接地原則延伸）。

---

### User Story 3 - 訂閱過濾（Priority: P2）

使用者可以維護自己的追蹤清單（公司代號）與通知類型偏好。管線派送前依訂閱過濾：沒訂閱的公司、關掉的類型不會進收件匣。預設只開 in-app 管道。

**Why this priority**: 防通知疲勞（PRD RK-1）。但在單一使用者、事件量小的 Phase 1 初期，先全收也能用——所以排在兩條 P1 之後。

**Independent Test**: 設定只追蹤 2330，餵 2330 與 2891 各一筆事件，驗證收件匣只有 2330 那則。

**Acceptance Scenarios**:

1. **Given** 使用者追蹤清單只有 2330，**When** 2330 與 2891 的事件各一筆進入管線，**Then** 收件匣只出現 2330 的通知。
2. **Given** 使用者關閉「新資料入庫」類型，**When** 一筆入庫事件進入管線，**Then** 不派送該通知。
3. **Given** 未設定任何訂閱（預設值），**When** 事件進入管線，**Then** 通知照常進收件匣（預設全收、不漏報）。
4. **Given** severity=alert 的合規/重大警示，**When** 即使該類型被關閉，**Then** 仍派送（安全攸關通知不可被靜音）。

---

### User Story 4 - 內部營運告警送 Slack（Priority: P2）

內部團隊（R1–R7）的營運通知——管線健康（ingestion 失敗、eval 掉分）、成本（token 預算逼近上限）、合規事故——除了進內部收件匣，也即時推送到團隊 Slack 頻道，取代「翻 log 才發現」。

**Why this priority**: PRD G3 / SC-NC-5（事故 5 分鐘內知悉）。依賴 US1 管線，但不阻塞分析師面的價值，故 P2。

**Acceptance Scenarios**:

1. **Given** 一筆 eval 分數跌破門檻的內部事件，**When** 進入管線，**Then** Slack 頻道收到一則含分數與門檻的訊息，且內部收件匣同步出現該通知。
2. **Given** Slack 派送失敗（網路暫時性錯誤），**When** 重試用盡仍失敗，**Then** 通知降級為僅收件匣，且失敗有記錄、通知不無聲丟失。
3. **Given** Slack 憑證未設定，**When** 內部事件進入管線，**Then** 管線照常運作（僅收件匣），不報錯、不外呼。

---

### User Story 5 - Digest 合併（Priority: P3）

同一公司、同一類型、同一天內的多則一般（info）通知合併為一則摘要通知（「2330 今日 3 則更新」），降低噪音。

**Why this priority**: 通知量大才有感；Phase 1 事件量小，是品質加分項。

**Acceptance Scenarios**:

1. **Given** 同日 2330 的 3 筆「新資料入庫」info 事件，**When** 依序進入管線，**Then** 收件匣最終呈現 1 則合併通知，計數為 3、保留各筆證據。
2. **Given** severity=watch 或 alert 的事件，**When** 進入管線，**Then** 不被合併、單獨成則（重要通知不可被摺疊稀釋）。

---

### Edge Cases

- 事件缺必填欄位（無識別碼、無標題）→ 拒收並記錄，不得讓壞事件弄垮管線。
- 對使用者通知的證據清單為空 → 不派送（US2 場景 4）。
- 合規審查服務本身失敗 → 退回確定性關鍵字底線（fail-to-floor），絕不因審查失敗而放行。
- 重複事件識別碼但內容不同 → 以第一筆為準、後到丟棄並記錄（exactly-once 優先於最新內容）。
- 收件匣為空時查詢未讀數 → 回 0，不報錯。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-NC-001**: 系統 MUST 定義統一通知資料格式，至少含：通知識別碼、來源事件識別碼、類型、受眾（使用者/內部）、公司代號（可選）、標題、摘要、嚴重度（info/watch/alert）、來源證據清單、詳情深連結、建立時間、已讀時間（可選）。
- **FR-NC-002**: 所有通知 MUST 經由單一通知服務的同一條「組裝 → 合規 → 訂閱過濾 → 派送」管線；事件生產者不得繞過管線直送任何管道。
- **FR-NC-003**: 受眾為使用者的通知文案（標題＋摘要）MUST 通過合規審查；被攔截者不得派送，且 MUST 自動產生一則內部合規事故通知。內部受眾通知不經攔截改寫但仍記錄。
- **FR-NC-004**: 受眾為使用者的通知 MUST 至少含 1 筆來源證據；證據為空者不得派送。
- **FR-NC-005**: 管線 MUST 以來源事件識別碼去重——相同識別碼僅產生一則通知。
- **FR-NC-006**: 使用者 MUST 能設定追蹤清單（公司代號集合）與通知類型開關；派送前 MUST 依訂閱過濾；未設定時預設全收；severity=alert 不受類型開關靜音。
- **FR-NC-007**: 收件匣 MUST 支援：未讀計數、列表（時間倒序）、標已讀、依公司代號與類型篩選。
- **FR-NC-008**: 內部通知 MUST 可推送到團隊即時訊息頻道；憑證 MUST 只存於環境設定（不入版控）；未設定憑證時管線 MUST 正常運作且不對外呼叫。
- **FR-NC-009**: 外送管道派送失敗 MUST 重試（沿用既有重試原則），用盡後降級為僅收件匣並記錄失敗，不得無聲丟失。
- **FR-NC-010**: 同一（公司代號、類型、日）的多則 info 通知 SHOULD 合併為一則 digest 通知，保留筆數與各筆證據；watch/alert 不合併。
- **FR-NC-011**: 缺必填欄位的事件 MUST 被拒收並記錄，不得中斷管線處理後續事件。
- **FR-NC-012**: 本功能 MUST NOT 修改既有 5 節點 workflow 的定義與行為；MUST NOT 在自動化測試中產生任何 LLM token 消耗或真實外部網路呼叫。

### Key Entities

- **NotificationEvent（事件）**: 生產者發出的原始事件——事件識別碼、類型、受眾、公司代號（可選）、標題、內文（不可信文字）、發生時間、證據。
- **Notification（通知）**: 管線組裝後、可派送的標準通知（見 FR-NC-001）；與事件 1:1 或 N:1（digest）。
- **Subscription（訂閱）**: 使用者的追蹤清單與類型開關；決定派送與否。
- **Channel（管道）**: 通知的出口介面——Phase 1 為收件匣與內部即時訊息；Email/LINE 為後續 phase。
- **ComplianceIncident（合規事故）**: 攔截事件的內部記錄通知，含被攔事件識別碼與攔截原因。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-NC-001**: 紅隊測試（誘導性事件樣本集）中，使用者收件匣出現的買賣建議字眼次數 = 0；每次攔截都有對應的內部事故通知（攔截記錄覆蓋率 100%）。
- **SC-NC-002**: 使用者收件匣中每則通知的來源證據覆蓋率 = 100%（無證據通知 0 則）。
- **SC-NC-003**: 重複事件去重正確率 = 100%（同識別碼事件重送 N 次，收件匣恆為 1 則）。
- **SC-NC-004**: 同一組事件樣本重跑 3 次，收件匣內容完全一致（確定性，對齊專題 SC-004 精神）。
- **SC-NC-005**: 自動化測試全程 0 token 消耗、0 真實外部呼叫；既有測試套件全數通過（不回歸）。
- **SC-NC-006**: 內部告警從事件進入管線到即時訊息送出 ≤ 5 分鐘（Phase 1 為同步派送，實測應 < 1 秒；上限為 PRD SC-NC-5 的承諾值）。

## Assumptions

- Phase 1 為單一使用者情境（無多人帳號體系，對齊 PRD OQ-3 暫緩）；訂閱設定為單一全域設定。
- 通知儲存為行程內（in-memory）收件匣＋可選檔案匯出；持久化（BigQuery 營運 dataset）為 Phase 2（PRD OQ-1）。
- 事件來源 Phase 1 以固定樣本（fixture）注入；真實生產者（Watchdog R3、ingestion R4、eval R5）之後以同一事件格式接上，管線不改。
- 合規審查沿用既有共用審查能力（確定性關鍵字底線＋可選智慧層），規則維護權在 R6。
- 即時訊息管道 Phase 1 僅內部 Slack；對外管道（Email/LINE）依 PRD 分期。
