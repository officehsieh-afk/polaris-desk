1216 2303 2357 2382 2412 2881 2882 2884 2886 2891 2892 3037 3231 3711 6669

## 前置依賴

- **先併 PR #69**（chunker 補 `--doc-type` / `--published-at`）。否則新 chunk 的
  `doc_type` / `published_at` 仍為 NULL，重蹈 PR #68 review 發現的 gap。
- 一把有效的 `GEMINI_API_KEY`（真 embedding 才可入庫；憲法不做假向量）。
- 來源文件：各檔法說會簡報 / 逐字稿（部分 IR 站擋 bot，TSMC 可、其餘不一）。

## 每檔流程

```bash
# 1. 抓法說會 簡報/逐字稿 PDF（fetch-tw-earnings-call 外掛）
#    → e.g. 2891_2025Q1_presentation.pdf

# 2. 切塊（帶 PR #69 新增的 metadata flags）
python -m polaris.ingestion.chunker 2891_2025Q1_presentation.pdf \
  --ticker 2891 --period 2025Q1 \
  --doc-type presentation --published-at 2025-04-17 \
  -o 2891_2025Q1.chunks.jsonl

# 3. 入庫（需 live GEMINI_API_KEY；寫進自己的 polaris_dev_<name>，不可寫 polaris_core）
python -m polaris.ingestion 2891_2025Q1.chunks.jsonl
```

`doc_type` 慣用值：簡報 = `presentation`、逐字稿 = `transcript`。

## 收斂

1. 逐檔 fetch → chunk → ingest 至 `polaris_dev_<name>`。
2. 在 `polaris_dev_<name>_staging` 彩排合併（同 PR #68 做法）。
3. 走 SOP §7 開後續 migration 併入 `polaris_core`（R1/R4 套用）。

## 邊界

- 寫 `polaris_core` 與 API 金鑰皆為人工 / R1-R4 把關，agent 不代為執行。
- 部分 IR 站擋 bot 時改走 MOPS 公開清單或人工下載。
