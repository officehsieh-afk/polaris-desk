# UX 痛點分析 — R3 串接成功後的使用流程

> 整理日期：2026-06-19｜撰寫：R7
> 基於 R3 需求清單、R7 Demo spec、專題憲法，推論所有端點串接成功後的使用者體驗，
> 定義潛在痛點並提出解法。前提：後端端點全數上線（GET /chunk、POST /contradiction、
> POST /peer-compare、GET /alerts with origin、GET /library、POST /subscriptions）。
>
> **修復進度**：✅ 痛點 2、4、5 已於 2026-06-19 完成前端修補（commit `841053c`）。

---

## 一、完整使用流程（串接後）

```
登入 Google OAuth
  └─ 研究助理（/research）
      1.  輸入研究問題（自然語言）
      2.  等待 POST /research 回傳
      3.  看 KPI 卡 + 摘要條列 + 毛利率趨勢圖（真實財務數字）
      4.  右側看 ReAct 思考追蹤（節點進度）
      5.  矛盾偵測自動觸發（POST /contradiction）→ 右側警示面板
      6.  點引用 chip → DocViewer 開啟原文（GET /chunk/{source_id}）
      7.  點「完整報告」→ 匯出 PDF（ReportModal）
      8.  點「送同業比較」→ 帶 query 跳頁

  └─ 同業比較（/peer）
      9.  輸入兩家公司 + 財報季別
      10. 等待 POST /peer-compare
      11. 看財務比較表 + 法說主題矩陣 + 趨勢折線圖
      12. 點引用 chip 驗證每個數字的來源

  └─ 監控 / 訂閱
      - 訂閱公司 → POST /subscriptions
      - 通知中心 GET /tracking-feed 追蹤最新動態

  └─ 歷史紀錄（/history）
      - 點擊還原上次研究結果（sessionStorage B 級還原）
```

## 二、使用者痛點 × 解法

---

### 痛點 1 — 驗證 10 個引用 = 開關 DocViewer 10 次，閱讀節奏被打斷

**發生在**：步驟 6（引用點擊）

**情境**

摘要有 5 個要點，每點帶 2 個引用 chip，分析師想逐一確認來源。
每次點 chip → DocViewer 右側展開 → 讀段落 → 關掉 → 回來繼續讀 → 再點下一個。
這個動作重複 10 次，主文閱讀節奏完全碎片化。

**對年長分析師的衝擊**

習慣在紙本畫線、做標記，不習慣在「讀」的途中強制切換「查」模式。

**解法建議**

- 引用 chip 加 **hover 展開小卡**：顯示 snippet 前兩行 + 頁碼 + trust score，
  不需開 DocViewer 就能快速掃描
- 只有需要讀完整原文時，才點進 DocViewer
- **前端可行**：chip 加 `title` tooltip 是最輕量起點；完整版用 Popover 元件，
  不需後端改動，GET /chunk 已串好

---

### 痛點 2 — 矛盾警示出現在右側欄，使用者正在看左側內容，兩者沒有視覺連結

> ✅ **已修復（2026-06-19，commit `841053c`）**

**發生在**：步驟 5（POST /contradiction 回傳後）

**情境**

`POST /contradiction` 回傳「全年指引 KPI 與摘要數字不一致」。
這條警示出現在右側「監控系統警示」面板，但使用者正在讀左側摘要第 3 點。
他完全不知道「第 3 點就是那個有矛盾的地方」，因為兩邊沒有視覺線索連結。

**對年長分析師的衝擊**

資深分析師對數字矛盾極度敏感，但若 UI 沒有明確指出「你正在看的這一行有問題」，
他很可能掃過去不察覺，或花大量時間手動對照。

**修復內容**

- `ContraAlert` 新增 `cite_key?: string` 欄位（`contraAlertStore.ts`）
- `buildMockContradictions` 帶入 `cite_key = revSum.cite`
- 摘要 bullet 若有對應矛盾警示，顯示 `.tag.mid`「矛盾」橘色標記（含 hover 說明）
- R3 `POST /contradiction` 上線後，帶入 `cite_key` 即可自動對應至正確 bullet

---

### 痛點 3 — 同業比較輸入欄要求打股票代碼，年長分析師習慣用公司名稱

**發生在**：步驟 9（同業比較輸入）

**情境**

分析師想比較「台積電」和「鴻海」，但輸入欄預期格式是 `2330`、`2317`。
他知道公司名字，不一定記得 4 位代號，且 UI 沒有提示或自動補全。

**對年長分析師的衝擊**

在 Bloomberg terminal 或 Excel，早就習慣輸入名稱後自動解析。
要求使用者記憶代碼，是強迫工作流程配合系統，而非系統配合使用者。

**解法建議**

- `GET /companies` 資料（R4 已上線）已有公司中文名 + ticker
- 輸入欄改為 **autocomplete combobox**：輸入中文公司名 → 下拉顯示 ticker + 名稱 → 選擇後自動填入
- **前端可行**：`useCompanies()` hook 已存在，改輸入元件即可，無需新端點

---

### 痛點 4 — 歷史紀錄還原的結果沒有「新鮮度」標示，分析師可能把舊資料當現況報告

> ✅ **已修復（2026-06-19，commit `841053c`）**

**發生在**：/history 點擊還原 → 研究助理頁

**情境**

分析師上週做了台積電的研究，今天點開歷史紀錄還原。
畫面顯示的 KPI 和摘要，外觀和「剛查的結果」完全一樣，
沒有任何「這是 7 天前的分析」提示。
如果他直接把這個結果帶進客戶報告，引用的是 7 天前的財務數字。

**金融風險**

台股公司的財報、法說、重大訊息隨時更新。舊資料 ≠ 當前事實，但 UI 沒有區分。

**修復內容**

- `history/page.tsx`：`sessionStorage` 存入 `time` 欄位（原始查詢時間）
- `research/page.tsx`：`restoredData` 存在時，結果頂部顯示 `.mock-note` 橘色警示條
  - 文字：`此為歷史分析（HH:MM），資料可能已有更新`
  - 附「重新查詢」按鈕，點擊清除還原資料並以同樣 query 重跑

---

### 痛點 5 — 匯出的 PDF 合規聲明不夠顯眼，分析師直接轉傳有 NFR-031 風險

> ✅ **已修復（2026-06-19，commit `841053c`）**

**發生在**：步驟 7（完整報告匯出）

**情境**

分析師覺得整理得很好，直接匯出 PDF 轉傳給交易部門或客戶。
但 PDF 裡若「非投資建議」只是小字頁尾，
收到的人（尤其非研究部門同事）可能忽略，把摘要當作投資指引。

**金融風險**

NFR-031 是憲法硬約束，不只要求 UI 不輸出買賣建議，
**對外流通的文件**同樣受此約束。

**修復內容**

- `ReportModal.tsx`：`report-content` 最頂部加紅框合規聲明區塊（html2canvas 捕捉範圍內）
  - 標題：`合規聲明 — NFR-031`（紅色粗體）
  - 正文：「本報告不構成任何投資建議、買賣建議或對特定有價證券之推薦」
  - 頁尾：報告產出時間戳記，資料以產出當下為準
- PDF 第一頁即可見，無法略過（非頁尾小字）

---

### 痛點 6 — ReAct 思考追蹤面板對分析師無意義，佔走右側欄有限空間

**發生在**：步驟 4（結果顯示後，右側面板）

**情境**

右側 `rcol-ctx` 面板顯示「理解查詢意圖 → 檢索文件庫 → 重排序候選 → elapsed_ms」
等技術步驟。這對工程師驗證系統行為有用，但對分析師是噪音。
更重要的「矛盾警示」和「引用追蹤器」也在同一個右側欄，被技術步驟擠壓。

**解法建議**

- ReAct trace 面板預設**收合**
- 摘要改為商業語言：
  `系統查閱了 12 份文件，涵蓋台積電 2026Q1 法說會逐字稿、合併財報`
- 技術細節（節點名稱、elapsed_ms）放在可展開的「進階」下，只給需要的人看
- 右側主要空間優先給「矛盾警示」和「引用追蹤器」
