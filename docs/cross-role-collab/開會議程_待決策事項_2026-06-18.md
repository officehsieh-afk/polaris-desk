## 議題 3：Alert 欄位命名不一致（先記錄，待討論）

> 本議題目前**不急著改**，但需要開會對齊，避免前端合併顯示時出 bug。

### 問題

同一個「監控警示面板」有兩個資料來源，欄位命名不同：

| 來源 | 嚴重度欄位 | 值 |
|------|-----------|-----|
| `GET /alerts` response | `severity` | `"alert"` / `"watch"` / `"info"` |
| `POST /contradiction` response | `level` | `"high"` / `"mid"` / `"info"` |

前端合併兩個來源顯示時，需要額外轉換，容易出錯。

### 選項

| 選項 | 說明 |
|------|------|
| 統一用 `severity`（`"alert"` / `"watch"` / `"info"`） | 語意對應顯示顏色（紅/橘/灰） |
| 統一用 `level`（`"high"` / `"mid"` / `"info"`） | 語意較通用 |

### 待確認

| # | 問題 |
|---|------|
| F | 統一用哪個欄位名稱與值？ |

## 各決策點背景需求

### A｜history 儲存後端

`polaris_core` 是唯讀共用庫（憲法約束），無法寫入 user-level 資料。目前 localStorage MVP 在同一瀏覽器可用，但換裝置或清快取後紀錄消失，不符合正式產品需求。這個決策會直接影響 B（誰來實作端點）和 E（訂閱清單用同一後端），建議優先確認。

---

### B｜history 端點由誰實作

`R2_需求清單_from_R7.md §2` 和 `R3_需求清單_from_R7.md §7` 都有出現這個端點，但沒有明確指定誰動工。若沒有人確認，/history 頁面（UI 已完成）會維持空白，localStorage 也無法升級為跨裝置版本。

---

### C｜workflow 自動寫入 history 可行性

`POST /history` 的觸發時機是 `/research` 或 `/peer-compare` 回應完成後。若後端 workflow 在結尾直接寫入，前端不需要額外一次 round-trip，前端 `api.ts` 也不需要加呼叫邏輯。若 workflow 無法自動寫入，前端需要主動呼叫，兩種做法的架構不同，需要先確認。

---

### D｜subscriptions/tracking-feed 由誰負責

目前三個端點（`GET/POST /subscriptions`、`GET /tracking-feed`）掛在 R3 清單，但功能是訂閱 CRUD 加上 `events` 表查詢，不含 AI 推理邏輯。R4 已有 `GET /events`、`GET /companies`、`GET /financials` 等讀端點，`GET /tracking-feed` 本質是其延伸。若沒有確認歸屬，通知中心的「訂閱設定」和「追蹤通知」兩個 tab 無法上線（前端 UI 已就緒）。

---

### E｜訂閱清單儲存後端

訂閱清單（per user 的 `tickers[]`）同樣不能存 `polaris_core`，性質與 history 相同。若 A 和 E 分開決定，可能之後需要做兩次遷移（例如 history 選 Firestore、訂閱又選 BQ）。建議與 A 同時確認，統一後端。

---

### F｜Alert 欄位統一命名

監控警示面板同時顯示 `GET /alerts`（欄位 `severity`：`"alert"` / `"watch"` / `"info"`）和 `POST /contradiction`（欄位 `level`：`"high"` / `"mid"` / `"info"`）兩個來源。前端合併顯示時需要轉換邏輯。若開發期間不對齊，上線後要前後端同時修改才能修正，越早確認越省成本。

---

### G｜`gics_classifications` 表是否存在（跨產業分析資料層）

Demo 場景 4（跨產業宏觀解析）明確依賴 `gics_classifications` 多產業 schema 做跨文件對齊。

**跨產業分析的定義確認**：這裡的「跨產業」指的是**同一家上市公司旗下的子公司 / 業群橫跨不同產業**（例如鴻海在電子製造、電動車、半導體等業群的佔比與宏觀背景），不是不同公司間的產業比較。

**目前 BQ 資料層缺口**：

| 表 | 內容 | 是否支援業群分拆 |
|----|------|----------------|
| `financial_metrics` | 公司整體合併數字 | ❌ 無子公司 / 業群層級拆分 |
| `chunks`（法說逐字稿） | 文字可能提及業群描述 | 半✅ 定性文字，非結構化數字 |
| `company_dim` | 母公司單一 `industry_id` | ❌ 無業群佔比 |
| `gics_classifications` | 場景 4 依賴的業群分類表 | ❓ 不在已知 schema |

前端業群佔比圖表需要子公司層級的結構化財務數字，目前無此資料。在 `gics_classifications` 或業群分拆資料確認存在前，前端只能呈現從法說文字 LLM 提取的定性描述（數字接地風險較高）。

**待確認**：`gics_classifications` 是否存在於 BQ？若不存在，業群分拆資料由誰補建、時程為何？

---

### H｜ColPali HTTP 端點由誰實作

`colpali_pages` 表已有 5,701 筆資料（20 個 ticker），但欄位表明確標注「暫無 HTTP 端點」，前端無法觸發查詢。Demo 場景 3 需要此端點展示多模態圖表解讀能力。端點本身是唯讀查詢，不含 AI 邏輯（類似 R4 的其他讀端點），但目前無人認領。

---

### I｜NFR-031 disclaimer 確認機制

Demo 場景草稿要求每次研究輸出結尾標「以下為事實摘要，非投資建議」，且驗收條件是「買賣建議 = 0」。目前不確定此 disclaimer 是由後端每次附在 response 裡，還是前端固定顯示。若兩者都沒做，demo 當天可能無法通過 NFR-031 審查，也缺乏讓觀眾看到的合規標示。

---

### K｜`POST /notifications/events` — 是否為新端點

R3 §2（`POST /contradiction`）備註中提到：「偵測到高/中風險時，是否會自動 push 到 NotificationService？還是需要前端另外呼叫 `POST /notifications/events`？」

這個端點在所有需求清單與跨角色協作清單中均未出現，但若矛盾偵測結果需要持久化到通知中心，它就是必要的。若無人負責，高風險矛盾只會在 session 內存在（sessionStorage），使用者關閉頁面後通知消失。

---

### L｜同業比較頁 PDF 匯出 + mock 資料的 NFR-031 合規確認

R7_前端待辦清單顯示「比較報告匯出 PDF ✅ 已完成」，但目前同業比較整頁顯示 hardcoded mock 財務數字（`POST /peer-compare` 尚未實作）。

問題：在 `POST /peer-compare` 上線前，使用者匯出的 PDF 內含的是假財務數字，外觀與真實分析報告完全相同。這份 PDF 一旦流出，可能被當成真實分析參考，觸及 NFR-031 合規邊界。

**短期過渡方案選項**（待確認）：
1. 在 `POST /peer-compare` 上線前停用匯出按鈕
2. 在匯出的 PDF 加上「示範資料，非真實分析」浮水印
3. 維持現狀，依賴使用者知悉為 demo 環境

---

### J｜Demo Watchdog 觸發機制

Demo 場景草稿的跨場景亮點是：上台前模擬 MOPS 公告 → Watchdog 自動觸發 → Alert Inbox 跳出警報。目前 `GET /alerts` 缺 `origin` 欄位（面板空白），且觸發機制（如何在 demo 當天模擬公告、Watchdog 的 cron / event-driven 切換）無任何規劃文件。此功能是 demo 的跨場景亮點，若未提前準備，demo 現場無法展示。
