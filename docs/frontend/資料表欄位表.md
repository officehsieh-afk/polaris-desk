> **資料來源**：直接由 live BigQuery schema + `__TABLES__` 列數 + 實際值統計產出（**2026-06-21 對帳**）。
> **權威庫**：`polaris-desk-team.polaris_core`（憲法預設 `VECTOR_BACKEND=bigquery`；pgvector 僅離線/Demo fallback）。
> **唯讀**：前端/一般帳號對 `polaris_core` 只有 READ，**不可寫**（寫入走 SOP §7 PR）。

> **🆕 2026-06-21 更新重點**（相對 2026-06-17 版）：

> 4. ⚠️ migration #132 的 `v_colpali_pages_semantic` View **已進 code、尚未套到 live BQ**（目前查不到），別先接。

## 1. `chunks` — RAG 文字片段（+ 向量）

| 欄 | 型別 | 可空 | 說明 |
|---|---|---|---|
| `chunk_id` | STRING | ❌ NOT NULL | 主鍵，片段唯一 ID |
| `ticker` | STRING | ✅ | 股票代號（join key） |
| `doc_type` | STRING | ✅ | 文件型別，可 filter（見下） |
| `fiscal_period` | STRING | ✅ | 財報期別，如 `2025Q4` |
| `published_at` | DATE | ✅ | 發布日（分區欄） |
| `chunk_text` | STRING | ✅ | 片段全文（顯示 / 引用接地用） |
| `embedding` | ARRAY\<FLOAT64\> | ✅ | 768 維向量（cosine）。**前端通常不用直接抓**，由後端 `/ask`、`/research` 做檢索 |
| `owner` | STRING | ✅ | 存取控制：擁有者帳號；NULL=公開 |
| `confidential` | BOOLEAN | ✅ | 機密旗標；查詢需帶 viewer 過濾 |

- `doc_type` 實際值：`major_news`(5,232)、`transcript`(1,608)、`news`(45)。
- ⚠️ **存取控制**：查 chunks 時後端會加 `(owner IS NULL OR owner=@viewer)` 與 `(NOT COALESCE(confidential,FALSE) OR owner=@viewer)`。前端不要自己拼 SQL 繞過，走後端 API。

## 2. `financial_metrics` — 結構化財務指標

| 欄 | 型別 | 說明 |
|---|---|---|
| `ticker` | STRING | 股票代號（join key） |
| `fiscal_period` | STRING | 期別，如 `2026Q2`、`2025Q4`（格式 `YYYYQn`） |
| `metric_id` | STRING | 指標代碼，可 filter（見下） |
| `value` | FLOAT64 | 數值 |
| `unit` | STRING | 單位：`千元`／`新台幣千元`／`新台幣元/股`／`%` |
| `source_id` | STRING | 來源 ID（引用接地） |
| `published_at` | DATE | 發布日 |

- `metric_id` 全集（14 種）：`revenue`、`revenue_yoy`、`revenue_delta`、`revenue_prior_year`、`gross_profit`、`operating_income`、`operating_expense`、`pretax_income`、`net_income`、`eps`、`ytd_revenue`、`ytd_revenue_prior_year`、`ytd_yoy`、`ytd_delta`。
- 一檔股票一期會有多列（每個 `metric_id` 一列）；前端做卡片/表格時用 `(ticker, fiscal_period, metric_id)` 當複合 key 取值。

## A. 🆕 語意 View（前端建議首選讀取面）

> 這三個 View 已把 ontology 標籤 JOIN 進事實表，**不含向量、不含 `owner`/`confidential` 機密欄**，
> 直接拿來顯示最省事。底層仍是 `polaris_core`；和 §1–§5 的原表是同一份資料、只是攤平加標籤。
> （優先順序：能走後端讀端點就走端點（§6）；要直讀 BQ 時，**用 View 而非原表**。）

### A-1. `v_company_profile` — 公司檔（20 列）

| 欄 | 型別 | 說明 |
|---|---|---|
| `ticker` | STRING | join key |
| `company_name` | STRING | 中文顯示名（如 `台積電`） |
| `english_name` | STRING | 英文名 |
| `market` | STRING | `上市`／`上櫃` |
| `industry_id` | STRING | 主產業代碼（如 `IND_FOUNDRY`） |
| `industry_name` | STRING | 主產業中文名（如 `晶圓代工`） |
| `is_financial` | BOOLEAN | 金融業旗標 |
| `aliases` | STRING | 逗號分隔別名（搜尋用） |
| `themes` | STRING | 逗號分隔主題標籤（如 `AI, 半導體`） |
| `secondary_industries` | STRING | 逗號分隔次要產業 |

→ 公司清單頁 / 搜尋自動完成 / filter 下拉「產業」「主題」的單一來源。

### A-2. `v_chunk_semantic` — 文字片段 ＋ 標籤（6,885 列）

涵蓋 §1 `chunks` 的可顯示欄（**省略 `embedding`/`owner`/`confidential`**），外加：
`company_name`、`industry_name`、`year`/`quarter`、`published_year`/`published_month`/`published_yyyymm`（時間 filter 好用）、
事件標籤 `event_type`/`event_type_name`/`event_subtype`/`event_category`/`event_severity`、
來源信任 `source_name`/`trust_tier`/`allowed_for_fact`/`citation_required`。

→ 引用接地卡片、可信度徽章、時間軸 filter 直接用。⚠️ 仍**建議走後端 `/ask`、`/research`** 做檢索（後端帶 viewer 過濾）；本 View 僅供讀展示，不要拿來繞過存取控制。

### A-3. `v_financial_metrics_semantic` — 財務指標 ＋ 中文標籤（4,620 列）

涵蓋 §2 `financial_metrics` 全欄，外加：
`company_name`、`metric_name`（指標中文名）、`metric_category`（指標分類）、
`year`/`quarter`/`month`、`period_start`/`period_end`（期間起訖日，畫圖用）、
`zero_tolerance`（零容忍旗標）、事件標籤 `event_type`/`event_type_name`。

→ 財務卡片 / 趨勢圖直接讀；`metric_name` 免自己維護代碼→中文對照。

## 6. 取數分層（決議：兩者都要）

| 資料 | 走哪裡 | 為什麼 |
|---|---|---|
| **語意問答 / 引用 / research / alert** | 後端 API（`POST /ask`、`POST /research`、`GET /alerts`，見 `src/polaris/api.py`） | 後端處理 embedding、**存取控制（owner/confidential）**、引用接地。前端不碰向量、不碰 `chunks` 原表 |
| **結構化表（公司清單 / 財務指標 / 事件流）** | **建議走穩定讀端點**：`GET /companies`、`GET /financials`、`GET /events`（2026-06-17 上線，見 `docs/API_使用指南.md`） | 前端耦合**API 契約**而非 physical BQ schema；後端已幫你 `JOIN company_dim`、clamp limit、固定回應形狀。底層仍是 `polaris_core`、**不碰 `chunks`** |
| **公司清單 / filter 字典 / 顯示標籤** | 若無對應端點 → **直接讀 §A View / §B r6 表** | View 已 JOIN 好標籤、無機密欄；r6 字典小可快取。比直讀原表安全省事 |
| **整頁視覺向量 `colpali_pages`** | 暫無 HTTP 端點 → 仍可**直接讀 BQ** | 進階視覺檢索用；尚未包成 API。用 `ticker`／`fiscal_period`／`doc_type` 當 filter。⚠️ migration #132 的語意 View `v_colpali_pages_semantic` 尚未套到 live，先讀原表 |

- **API 契約看這裡，別抄**：FastAPI 自動產生，永不過期 —— `GET /openapi.json`、互動文件 `GET /docs`；人讀導覽見 `docs/API_使用指南.md`。前端建議用 `openapi-typescript` 從它生 TS 型別，零漂移。
- **本欄位表的用途**：理解每張表「有哪些欄、能 filter 哪些值」；實際取數請優先走上面的讀端點（結構化）或 `/ask`・`/research`（語意），少數尚未包成 API 的（`colpali_pages`）才直讀 BQ。
- **顯示用公司中文名**：JOIN `company_dim USING (ticker)`（已 live）。
- **每個數字都要有來源**（憲法引用接地）：帶上 `source_id`／`source_url`／`chunk_id`。
- **🔴 NFR-031**：任何畫面都不得輸出買賣建議。
- ⚠️ **`chunks` 不要前端直讀**：有 `owner`／`confidential` 存取控制，必須由後端帶 viewer 過濾。需要文字片段就走 `/ask`、`/research`。

### 本檔會不會過期？自己驗一下（單一事實來源＝ live BQ）
本表是 2026-06-21 的快照。要確認當下 schema / 列數，直接跑（不需寫權限）：
```sql
-- 欄位
SELECT table_name, column_name, data_type, is_nullable
FROM `polaris-desk-team.polaris_core.INFORMATION_SCHEMA.COLUMNS`
ORDER BY table_name, ordinal_position;
-- 表/View 清單
SELECT table_name, table_type
FROM `polaris-desk-team.polaris_core.INFORMATION_SCHEMA.TABLES`
ORDER BY table_type, table_name;
```
若與本檔有出入，**以 `INFORMATION_SCHEMA` 為準**並回頭更新本檔。
